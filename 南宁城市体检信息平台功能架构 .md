# 南宁城市体检信息平台功能架构 

# 一、项目目标

构建一个基于 WebGIS 的城市体检信息平台，用于支撑城市更新片区的数据查询、体检指标管理、自动化评分分析、问题诊断识别、更新潜力评估、主观意愿反馈和后续更新决策辅助。

平台目标包括：

- 城市更新片区数据查询；

- 城市各维度体检数据管理；

- 多维度指标自动化评分分析；

- 诊断问题识别；

- 更新潜力评估；

- 留改拆研判；

- 多主体主观意愿反馈；

- 更新路径与方案决策辅助。

# 二、平台总体定位

平台定位为“基于 WebGIS 的城市体检评估与城市更新决策辅助平台”。平台不是普通数据看板，而是围绕地块、建筑、小区、社区、城中村、老旧商业区、老旧厂区和城市基础设施等空间对象，形成“数据录入—指标计算—综合分析—地图展示—意愿反馈—决策辅助”的全过程交互平台。

平台应同时支撑客观体检数据分析与主观意愿反馈分析：客观部分通过指标体系、权重模型和计算公式形成评分结果；主观部分通过不同主体的候选意愿调查、自由意见填报和大模型 API 归纳总结，辅助管理者理解多方诉求。

**平台总体业务闭环为：**

基础空间数据 → 指标体系配置 → 原始数据导入 → 指标数据转换 → 单项指标计算 → 综合分析计算 → GIS地图展示 → 主观意愿调查 → 大模型意见整理 → 统计分析导出 → 更新决策辅助。

# 三、技术架构建议

平台采用 Web 端应用架构，建议由前端应用、后端服务、指标计算服务、空间数据库、文件存储服务和权限认证服务共同组成。

|**类别**|**建议技术/能力**|**说明**|
|---|---|---|
|前端|React、TypeScript、Vite、Ant Design、Zustand/Pinia|用于构建 Web 端交互界面、地图页面、后台管理页面和表单配置页面。|
|地图展示|OpenLayers / Mapbox GL / Leaflet|用于前端 WebGIS 地图展示、图层控制、图元点击查询和空间对象可视化。|
|统计图表|ECharts|用于指标统计、反馈统计、综合分析结果图表展示，也可用于部分轻量级地图可视化。|
|后端服务|Node\.js \+ NestJS|用于用户、权限、数据、指标、调查、文件、日志等业务管理。|
|计算服务|Python FastAPI|用于单项指标计算、综合分析计算、空间计算、模型计算和大模型 API 调用封装。|
|空间数据处理|GeoPandas|用于后端空间数据清洗、空间叠加、指标预处理和空间分析；不建议作为前端地图展示库。|
|数据库|PostgreSQL \+ PostGIS|用于存储空间对象、指标体系、指标值、评分结果、用户反馈和操作日志。|
|数据格式|GeoJSON、Polygon、Excel、CSV、图片、PDF|支持空间数据、属性数据、调研数据和附件材料的导入导出。|

# 五、用户注册与权限管理模块

## 5\.1 功能定位

该模块用于控制谁可以进入平台、谁可以查看哪些内容、谁可以提交哪些反馈、谁可以管理指标和数据。平台不采用完全开放注册，而采用“白名单注册 \+ 身份绑定 \+ 后台审核 \+ 权限访问”的机制。

## 5\.2 白名单注册功能

平台不面向所有公众开放注册，只有被提前录入白名单的用户才允许注册。白名单由系统管理员或体检管理员在后台维护。

|**白名单字段**|**说明**|
|---|---|
|手机号|用于注册前置校验、验证码登录和账号识别。|
|姓名|用于身份比对和后台审核。|
|用户类型|可包括系统管理员、居民、政府管理者、市场主体等。|
|关联街区|限定用户所属或可访问的街区范围。|
|关联地块|限定用户绑定的地块对象。|
|关联住址|可绑定小区、楼栋、门牌、住址或空间对象编号。|
|可访问范围|定义用户可以查看的地图范围、指标范围和数据范围。|
|是否允许反馈|定义用户是否可以参与主观意愿调查。|
|是否允许管理|定义用户是否可以进入后台管理或配置页面。|
|账号状态|未注册、待审核、已通过、已禁用。|

**注册流程为：**

1. 输入手机号；

2. 系统判断手机号是否在白名单中；

3. 白名单校验通过后进入注册页面；

4. 填写基础信息；

5. 通过下拉菜单绑定街区、地块、小区、楼栋或住址；

6. 可选上传认证材料；

7. 提交后台审核；

8. 审核通过后开通账号并授予对应访问权限。

## 5\.3 注册信息填写要求

|**信息类别**|**内容**|
|---|---|
|基础账号信息|手机号、验证码、登录密码（可选）、邮箱（可选）。|
|基本身份信息|姓名、联系方式、用户类型。|
|空间绑定信息|街区、地块、小区、楼栋、住址或对象编号。|
|认证材料|可选上传产权证明、居住证明、单位证明、营业执照、授权材料等。|
|注册状态|待审核、审核通过、审核驳回、信息需补充。|

街区、地块、住址等信息不建议让用户自由填写，应通过后台维护的标准对象库以下拉菜单方式选择，避免数据不一致。

## 5\.4 登录功能

|**登录方式**|**说明**|
|---|---|
|手机验证码登录|作为主要登录方式，便于白名单校验和用户身份识别。|
|微信绑定登录|用户首次绑定微信后，可通过微信快捷登录。|
|账号密码登录|作为后台管理端或备用登录方式。|
|登录状态保持|用户短期内无需重复登录。|
|退出登录|支持主动退出平台。|
|登录异常提示|对白名单不存在、审核未通过、账号被禁用等情况进行提示。|

登录逻辑为：手机号验证码登录 → 判断是否为白名单用户 → 判断是否审核通过 → 判断账号是否启用 → 判断访问权限 → 进入对应页面。

## 5\.5 身份验证与访问权限

平台应设置身份验证机制和访问权限控制。身份验证用于确认用户是否与某个街区、地块、小区、建筑或单位存在关联；访问权限用于控制用户可以查看和操作哪些内容。

|**权限维度**|**控制内容**|
|---|---|
|空间范围权限|控制用户可以查看哪些街区、地块、小区、建筑、社区或基础设施。|
|数据内容权限|控制用户可以查看哪些指标、评分、综合分析结果、反馈统计和报告。|
|操作权限|控制用户是否可以新增、修改、删除、审核、导入、导出数据。|
|反馈权限|控制用户是否可以参与调查、点赞点踩、打分、排序和提交文字意见。|
|管理权限|控制用户是否可以配置指标、权重、公式、白名单和调查问卷。|
|发布权限|控制用户是否可以发布主观意愿调查、绑定对象和筛选调查对象。|
|**用户类型**|**平台结构中预留的功能方向**|
|系统管理员|平台配置、白名单管理、权限管理、指标管理、数据管理、模型管理和系统维护。|
|居民|查看绑定对象相关信息，参与主观意愿调查，提交个人反馈。|
|政府管理者|查看管理范围内的体检数据、综合分析结果、反馈统计和报告成果。|
|市场主体|在授权范围内查看相关地块或项目数据，提交参与意愿、合作诉求或更新建议。|

# 六、GIS 地图主界面模块

## 6\.1 功能定位

GIS 地图主界面是平台核心页面，用于展示空间对象、指标评分结果、综合分析结果和主观反馈结果。用户进入平台后，可基于地图完成片区浏览、图层切换、指标查询、图元点击、详情查看和结果统计。

## 6\.2 功能目标

- 片区展示；

- 地图切换；

- 点击查询；

- 评分热力图；

- 图层控制；

- 单项指标地图展示；

- 综合分析地图展示；

- 主观意愿反馈结果展示。

## 6\.3 地图展示类型

|**地图类型**|**展示内容**|
|---|---|
|单项指标地图|用于展示某一个具体体检指标的空间分布，例如停车泊位缺口数、养老服务设施覆盖率、建筑质量、建筑年代等。|
|综合分析地图|用于展示多个指标计算后的综合结果，包括综合评分、更新潜力分析、留改拆研判、问题诊断、主观意愿反馈热力图、意愿支持率分布图等。|

需要明确：更新潜力分析和留改拆研判均属于综合分析，不应与综合分析并列为同级计算模块。计算服务层应表达为“单项指标计算 \+ 综合分析计算”。

## 6\.4 地图交互逻辑

**指标查询交互流程：**

- 选择空间对象维度 → 选择一级维度 → 选择二级维度 → 选择体检指标 → 点击查询 → 地图显示对应评分热力图。

**图元点击交互流程：**

- 用户点击地图上的片区、地块、建筑、小区、社区、老旧商业区、老旧厂区、城中村/自建房或城市基础设施图元后，系统弹出右侧详情栏。

**右侧详情栏展示内容包括：**

- 图元类型；

- 图元编号；

- 所属街区、地块、小区、社区；

- 从属关系；

- 基础信息；

- 原始数据；

- 指标数据；

- 单项指标评分；

- 综合分析结果；

- 问题诊断与更新建议；

- 更新潜力分析；

- 留改拆研判；

- 主观意愿反馈统计结果。

## 6\.5 地块颜色规则

- 规则一：可根据不同指标阈值自定义颜色，例如高风险、中风险、低风险分别对应不同颜色。

- 规则二：可根据不同对象类型自定义颜色，例如住宅、商业、工业、基础设施等类型分别设色。

- 规则三：综合分析地图可根据综合评分、更新潜力、留改拆结果或主观意愿支持率进行分级设色。

## 6\.6 各维度图层

- 片区图层；

- 社区图层；

- 小区图层；

- 建筑图层；

- 城中村图层；

- 商业街区图层；

- 老旧厂区图层；

- 基础设施图层；

- 主观反馈点位图层；

- 综合分析结果图层。

## 6\.7 可展开/收缩侧边栏

为避免页面信息一次性全部列出，GIS 主界面应采用可展开/收缩侧边栏设计。

|**页面区域**|**设计要求**|
|---|---|
|左侧功能栏|默认收起，仅显示功能图标；点击后展开指标体系、图层控制、综合分析、主观反馈和新增指标入口。|
|右侧详情栏|默认隐藏；用户点击地图图元后展开，内部采用 Tab 页切换。|
|底部统计栏|可折叠展示当前地图结果的统计图、指标分布、反馈统计和导出入口。|

右侧详情栏建议设置以下 Tab：基础信息、指标评分、综合分析、问题诊断、主观反馈、更新建议。

# 七、体检指标体系管理模块

## 7\.1 功能定位

体检指标体系管理模块用于维护平台中的所有体检指标，包括现有指标、新增体检指标、新增综合评估维度、指标单位、指标权重、指标规则和颜色规则。

## 7\.2 指标体系层级

指标体系采用以下层级：

更新改造任务 → 空间对象类型 → 一级维度 → 二级维度 → 体检指标 → 原始数据 → 指标数据 → 单项评分 → 综合分析结果。

|**层级**|**说明**|**示例**|
|---|---|---|
|更新改造任务|面向城市更新工作的任务类别。|既有建筑改造利用、完整社区建设、城市基础设施建设改造等。|
|空间对象类型|指标挂接的空间对象。|地块、建筑、小区、社区、老旧商业区、老旧厂区、城中村、基础设施。|
|一级维度|体检指标的一级分类。|建筑、小区、社区、城市基础设施、文化特色等。|
|二级维度|体检指标的二级分类。|安全耐久、空间利用、公共配套、市政设施、道路交通等。|
|体检指标|具体可计算或可填报的指标项。|存在结构安全隐患住宅数量、停车泊位缺口数、消防站服务覆盖率等。|
|原始数据|填报、导入或采集的基础数据。|建筑年代为1995年、停车位数量为100个。|
|指标数据|原始数据经过清洗和转换后的指标值。|停车缺口率、建筑年代等级、设施覆盖率等。|
|单项评分|某一指标按照规则计算后的得分。|停车缺口评分、建筑质量评分等。|
|综合分析结果|多个指标综合计算后的结果。|综合评分、更新潜力、留改拆研判、问题诊断。|

## 7\.4 新增体检指标

平台允许动态新增体检指标。新增体检指标时，需要完成从指标定义、数据导入、评分规则、权重配置、地图展示到综合分析的完整配置。

|**配置项**|**内容**|
|---|---|
|指标名称|新增体检指标名称。|
|所属空间对象|地块、建筑、小区、社区、基础设施等。|
|所属维度|一级维度、二级维度。|
|数据类型|数值、文本、比例、布尔、枚举、空间数据等。|
|单位|个、栋、米、平方米、%、分等。|
|数据来源|街道填报、GIS导入、空间计算、系统录入、接口获取等。|
|是否参与单项评分|确定该指标是否需要被转换为单项得分。|
|是否参与综合分析|确定该指标是否进入综合评分、更新潜力、留改拆或问题诊断模型。|
|评分规则|阈值评分、公式评分、等级评分、一票否决、人工评分等。|
|权重系数|可自定义修改，用于综合分析计算。|
|颜色规则|按指标阈值设色或按对象类型设色。|
|输出方式|地图、表格、统计图、报告等。|

## 7\.5 新增综合评估维度

除新增单个体检指标外，平台还应支持新增综合评估维度。综合评估维度用于组织多个体检指标，形成综合评分、更新潜力、留改拆研判、问题诊断等综合分析结果。

新增综合评估维度时，需要配置：

- 综合评估维度名称；

- 关联的体检指标；

- 各指标权重；

- 计算公式；

- 定性规则；

- 结果分级规则；

- 地图颜色规则；

- 是否用于更新潜力；

- 是否用于留改拆研判；

- 是否用于问题诊断；

- 是否纳入报告导出。

例如：新增综合评估维度“更新潜力”，可关联建筑质量、建筑年代、容积率、建筑密度、设施短板、内涝风险、规划一致性等指标，并通过“更新潜力值 = Σ（指标得分 × 权重系数）”计算。

## 8\.3 数据上传与校验

街道或相关主体填写 Excel 后，平台应支持上传并自动校验。校验内容包括：

- 字段是否匹配；

- 编号是否重复；

- 必填项是否缺失；

- 数据格式是否正确；

- 单位是否一致；

- 坐标或空间对象是否能匹配；

- 是否存在异常值；

- 是否存在重复记录；

- 是否能与地块、建筑、小区、社区等空间对象绑定。

## 8\.4 三类数据结构

平台需明确区分原始数据、指标数据和综合评估数据。

|**数据类型**|**说明**|**示例**|
|---|---|---|
|原始数据|街道、管理员、用户填报或导入的基础信息。|建筑年代为1995年，停车位数100个，建筑质量为一般。|
|指标数据|原始数据经过清洗、转换、标准化后的体检指标值。|建筑年代等级、停车缺口率、建筑质量评分。|
|综合评估数据|多个指标通过权重和公式综合计算后的结果。|综合评分、更新潜力值、留改拆结果、问题诊断结果。|

数据流为：原始数据导入 → 数据清洗 → 指标数据转换 → 单项指标评分 → 综合分析计算 → 地图展示 / 报告导出。

# 九、指标计算与综合分析模块

## 9\.1 功能定位

指标计算与综合分析模块是平台的核心计算层。新版架构中，计算服务层应划分为“单项指标计算”和“综合分析计算”两部分。其中，更新潜力分析和留改拆研判均属于综合分析计算。

## 9\.2 单项指标计算

单项指标计算用于将某一个指标转换为可比较、可展示、可统计的得分或等级。

|**计算方式**|**说明**|
|---|---|
|手动录入|由管理员或街道工作人员直接填入指标值或评分。|
|公式计算|根据预设公式自动计算，例如 green\_area / parcel\_area。|
|GIS 空间计算|通过空间叠加、缓冲区、覆盖率、距离分析等方式计算指标。|
|Excel 导入|导入标准模板后直接生成指标数据。|
|外部接口接入|后续可接入政务数据接口或其他业务系统。|
|人工修正|允许管理员对自动计算结果进行人工校正并记录原因。|

示例：绿地覆盖率 = green\_area / parcel\_area；停车泊位缺口率 = 停车泊位缺口数 / 停车需求数；建筑年代得分 = 根据建筑建成年代分级赋分。

## 9\.3 综合分析计算

综合分析计算是在多个单项指标结果基础上的进一步分析，主要包括综合评分、更新潜力分析、留改拆研判和问题诊断。

|**综合分析类型**|**说明**|
|---|---|
|更新潜力分析|根据建筑质量、建筑年代、空间利用、设施短板、风险隐患、规划一致性等指标计算更新潜力。|
|留改拆研判|结合定性一票否决规则和定量权重评分，形成保留、改造、拆除等初步判断。|
|问题诊断|根据指标阈值、异常数据和规则判断形成问题清单。|

更新潜力值可配置为：更新潜力值 = Σ（相关指标得分 × 对应权重系数）。

## 9\.4 定性与定量规则

平台综合分析应同时支持定性规则和定量规则。

|**规则类型**|**说明**|**示例**|
|---|---|---|
|定性规则|一票否决或强制判断规则，优先级高于普通权重计算。|存在严重结构安全隐患时，优先进入重点改造或拆除研判；涉及保护要求时，不允许拆除。|
|定量规则|通过指标权重、分值和公式计算形成结果。|潜力值高则优先改造或拆除重建，潜力值中则综合整治，潜力值低则保留维护。|

## 9\.5 权重与系数管理

平台应支持选择对应指标设置权重，并支持系数自定义修改。

|**配置内容**|**说明**|
|---|---|
|指标权重|某个体检指标在综合分析中的权重。|
|维度权重|某一维度在综合评分中的权重。|
|系数设置|更新潜力计算或其他综合模型中的可调参数。|
|一票否决规则|定性判断条件和优先级。|
|公式配置|支持自由配置综合评分公式和潜力计算公式。|
|阈值配置|支持设置结果分级标准。|
|版本保存|保存不同权重、公式和模型版本，便于追溯。|

## 9\.6 计算公式自由度

平台应提供较高的计算公式配置自由度。管理员可以配置加法、减法、乘法、除法、条件判断、阈值判断、指标引用、权重引用和一票否决规则。

# 十、主观意愿反馈模块

## 10\.1 功能定位

主观意愿反馈模块用于采集不同主体对更新方案、体检问题和空间对象的主观意见。新版平台中，主观意愿反馈不再只是简单意见收集，而应支持“候选意愿配置—对象筛选—绑定发布—用户填报—统计汇总—大模型整理”的完整流程。

## 10\.2 候选意愿选项配置

体检管理员可以在后台为不同主体配置候选意愿选项，并针对不同主体绑定发布。

管理员发布调查时，应能够筛选调查对象。筛选条件包括：

- 用户类型；

- 街区；

- 地块；

- 小区；

- 建筑；

- 住址；

- 白名单分组；

- 是否已认证；

- 是否已绑定对象；

- 是否已参与过相关调查。

示例：只向某一地块关联居民发布调查；只向某一街区市场主体发布合作意愿调查；只向某一小区住户发布加装电梯意愿调查。

|**反馈形式**|**说明**|

|排序|对优先解决事项进行排序。|
|单选/多选|从候选更新选项中选择。|
|文字说明|填写详细主观意见。|
|图片上传|上传现场照片或证明材料。|

## 10\.5 意愿反馈记录内容

# 十一、大模型意见整理模块

## 11\.1 功能定位

平台应调用大模型 API，对大量主观文字意见进行自动整理、分类和总结。大模型 API 用于处理用户提交的自然语言意见，帮助管理员快速识别高频问题、分歧点、支持倾向和方案偏好。

需要明确：大模型只用于主观意见的整理、归纳和辅助分析，不直接替代评分模型，也不直接决定最终更新方案。

## 11\.2 处理内容

- 居民文字意见；

- 市场主体反馈；

- 政府管理意见；

- 方案建议；

- 问题描述；

- 投诉与纠错；

- 更新偏好说明；

- 图片或位置反馈对应的文字说明。

## 11\.3 输出结果

|**输出内容**|**说明**|
|---|---|
|意见摘要|自动总结主要观点。|
|高频问题|提取用户反复提到的问题。|
|情绪倾向|识别支持、反对、中立等倾向。|
|关注重点|识别停车、消防、电梯、环境、搬迁、费用等关注点。|
|方案偏好|判断用户更倾向哪类更新方式。|
|分歧点|识别不同主体之间的意见冲突。|
|风险提示|提示可能影响推进的问题。|
|管理建议|为管理员提供后续处理建议。|

## 11\.4 大模型处理流程

用户提交主观意见 → 系统存储原始意见 → 调用大模型 API → 自动分类与摘要 → 生成结构化标签 → 汇总统计结果 → 展示给管理员和决策者。

# 十二、统计分析与成果导出模块

## 12\.1 功能定位

统计分析与成果导出模块用于汇总平台中的指标计算结果、综合分析结果和主观意愿反馈结果，并支持图表展示、Excel 导出和报告输出。

## 12\.2 统计内容

|**统计类型**|**内容**|
|---|---|
|指标统计|各类体检指标评分分布、指标值分布和异常指标统计。|
|综合分析统计|综合评分、更新潜力、留改拆结果、问题诊断结果统计。|
|问题统计|问题类型、问题数量、问题等级、问题空间分布。|

# 十五、平台数据流与业务流程

平台完整业务流程如下：

1. 管理员维护街区、地块、建筑、小区、社区和基础设施等空间对象；

2. 管理员维护白名单用户和访问权限；

3. 用户通过手机号或微信绑定登录，并绑定街区、地块或住址；

4. 平台导出标准 Excel 模板，供街道或相关部门填写；

5. 街道上传填报后的 Excel 数据；

6. 系统完成数据校验、原始数据入库和指标数据转换；

7. 系统根据单项指标规则计算单项指标得分；

8. 系统根据综合分析模型计算综合评分、更新潜力、留改拆研判和问题诊断；

9. GIS 地图展示单项指标地图和综合分析地图；

10. 体检管理员配置主观意愿调查，筛选调查对象并绑定发布；

11. 用户提交点赞点踩、打分、排序、文字意见和图片反馈；

12. 系统汇总主观反馈结果，并调用大模型 API 进行意见整理；

13. 平台生成统计图表、反馈摘要、问题清单和导出报表；

14. 管理者结合客观体检结果、综合分析结果和主观意愿结果进行更新决策辅助。
